# Partsone バックエンド研修課題リポジトリ

## 取り組んだ内容
### 設計の基本課題
1. API設計1
    - エクセルテンプレートを参考にして、エクセル上で簡単なAPI設計書を作成しました。ここで作成したエクセルファイルは.docsフォルダに保存されています。なお、途中で色々更新したのですが、その時に更新したのはyamlファイルおよびER図なので、このエクセルファイルは最新版ではないです。

2. API設計2
    - API設計1で作成したエクセルファイルを参考にして、自分でyamlファイルを作成しました。作成したyamlファイル(my-openapi.yaml)は.docsフォルダの中のapiフォルダに保存されています。

3. DB設計
    - 作成したyamlファイルを参考に、dbdiagram.ioを使ってER図を作成しました。作成したER図は.docsフォルダの中のdbフォルダに保存されています。現在二つ保存されていますが、日付が新しい方が最新版となります。

### 実装の基本課題
1. APIの実装
    - マイグレーションファイルの作成： ArticleとCommentの2つのテーブルを作成し、その中にカラムや制約の定義をしました。
    - APIエンドポイントの実装: routes/api.phpにGET, POST, PUT, DELETEの4つのエンドポイントを実装しました。
    - コントローラの作成: /app/Http/Controllers/Api/CommentController.phpにGET, POST, PUT, DELETEの4つのエンドポイントに対応するメソッドを作成しました。今回DELETE機能は、データベースから情報を消去する物理消去を採用しました
    - Resourceファイルの作成: /app/Http/Resources/CommentResource.phpにCommentに関するAPIリソースの定義を行いました。
2. Seederの実装
    - Factoryファイルの作成: /database/factoriesの中のFactory.phpファイルに、ArticleとCommentのデータの雛形の定義を行いました
    - Seederファイルの作成: /database/seeders/の中のSeeder.phpファイルでは実際にどのようにサンプルデータを作成するのかの定義を行いました
    - 動作確認: VSCodeのThunder Clientを使って、各エンドポイントが正しく動作するかどうかを確認しました。

### 実装の際の工夫点、課題点
- 今回取り組んだ基本実装では、ユーザーの認証機能の実装が必要でなかったため、データベースはArticleとCommentの二つのテーブルのみとし、Userテーブルは作成しませんでした。
- しかしその結果、"自分が投稿したコメントのみを削除・更新できる"という要件を満たすことが現状できていません。ユーザー認証を行わずにこの要件を満たす方法として、コメントの投稿の際にパスワードの設定を義務付けて削除・更新の際にパスワードを要求するという方法を考えましたが、まだ実際に実装はしていません。
- CommentテーブルとArticleテーブルで共通してArticle_idというカラムを作成しようと思いましたが、テーブルの主キーは単純な"id"の方が良いという記述を見たので、Articleテーブルの主キーは"id"とし、Resourceファイルの中でidをarticle_idに変換して返すようにしました。
